# 项目规范化与部署准备设计

## 背景

当前项目是一个个人维护的 AI 旅行规划应用，前端为 React 19 + Vite，后端为 Node.js + Express，已经具备基础功能、认证、AI 流式响应、RAG 检索和分享能力。项目即将部署到阿里云 ECS，域名正在备案中。

本次目标不是做企业级大重构，而是把项目整理到适合个人长期维护、作品集展示和单机部署上线的规范程度。

## 目标

1. 保持当前前后端分离结构，避免引入过重的 workspace 或 Docker 复杂度。
2. 优先提升接口、状态、错误处理和部署流程的清晰度。
3. 让项目可以较稳定地部署到阿里云 ECS：前端静态资源由 Nginx 托管，后端 Node 服务由 PM2 守护。
4. 让 README、环境变量示例和部署说明能真实反映项目，适合作品集展示。
5. 每个阶段完成后执行检查，并在最后给出评估和下一轮优化建议。

## 非目标

1. 不在本轮改成 pnpm workspace。
2. 不在本轮强制全量 TypeScript 迁移。
3. 不在本轮引入 Docker Compose。
4. 不在本轮重写 RAG 或更换数据库。
5. 不在本轮引入完整测试体系，只保留后续扩展空间。

## 推荐路线

采用“个人项目标准化 + 可部署优化”的轻量路线：

- 前端维持 Vite SPA 架构。
- 后端维持 Express ES Modules 架构。
- 先规范 API 边界、SSE 流程、错误处理和部署文档。
- TypeScript 采用渐进式迁移，优先从共享类型、API、hooks、stores 开始。
- 部署优先使用 Node + pnpm + PM2 + Nginx，后续需要时再容器化。

## 阶段设计

### 阶段一：工程入口规范

目标：让项目对自己和外部查看者都更清楚。

改动范围：

- 更新 README，使其说明真实项目能力、技术栈、本地启动、环境变量、构建与部署方式。
- 增加 `.env.example`，列出后端运行所需配置。
- 补充部署说明，说明阿里云 ECS 上的推荐目录结构、前端构建、后端启动、Nginx 反代、PM2 守护和健康检查。
- 检查 `package.json` 脚本，确保本地检查命令清晰可用。

验收：

- 新人根据 README 可以理解项目并启动。
- 部署文档能指导阿里云 ECS 首次上线。
- 不暴露真实密钥。

### 阶段二：前端接口规范

目标：降低页面、store、hook 对接口细节的直接依赖。

改动范围：

- 梳理现有 `src/api` 或新增统一 API 层。
- 统一请求封装：JSON 请求、认证 token、错误解析、401 处理。
- 将认证、天气、分享、聊天、推荐等接口按领域拆分。
- 保持调用方改动最小，不做无关 UI 重构。

验收：

- 页面和 store 不再散落重复的 fetch 逻辑。
- API 错误返回有统一处理方式。
- `pnpm check` 通过，核心页面可正常调用接口。

### 阶段三：流式请求规范

目标：提高 AI 推荐和聊天流式输出的稳定性。

改动范围：

- 增强 `useSSE`：支持 `AbortController`、重复请求保护、错误事件处理、结束状态复位。
- 明确前后端 SSE 事件类型：`step`、`chunk`、`notice`、`complete`、`error`。
- 保证用户快速重复提交时不会出现多个流互相写状态。
- 对 JSON 解析失败和网络中断提供友好错误。

验收：

- 可以主动取消或自动终止旧请求。
- 异常流不会让页面长期卡在 loading。
- 推荐和聊天的流式体验保持正常。

### 阶段四：后端接口规范

目标：让后端错误、参数和响应更可控。

改动范围：

- 增加轻量参数校验工具或引入 zod，优先覆盖登录注册、聊天、推荐、分享接口。
- 增加统一错误处理中间件。
- 统一常见错误响应格式。
- 保持现有 Mock/降级逻辑不被破坏。
- 对 chat/recommend 增加输入长度保护，避免异常长 prompt 带来成本和稳定性风险。

验收：

- 非法参数得到明确 400 响应。
- 未捕获异常不会直接暴露堆栈给前端。
- `server` 侧语法检查通过。

### 阶段五：部署规范

目标：让项目能清晰地部署到阿里云 ECS。

改动范围：

- 增加 PM2 示例配置。
- 增加 Nginx 反代示例：前端静态资源、`/api` 代理到后端、SPA fallback。
- 补充生产环境检查清单：Node 版本、pnpm、环境变量、端口、安全组、防火墙、日志、健康检查。
- 说明备案完成前后域名和 IP 访问的差异。

验收：

- 文档覆盖从构建到启动、反代、排错的基本流程。
- 用户可以按文档在 ECS 上部署首版。

## 数据与接口边界

前端应通过统一 API 层访问后端，不直接在页面中拼接 fetch 细节。后端继续以 `/api/*` 暴露服务，Vite 开发环境通过 proxy 转发，生产环境由 Nginx 转发。

推荐保留的边界：

- `src/api/*`：接口调用、错误处理、认证头。
- `src/hooks/*`：面向组件的交互逻辑，比如 SSE 和天气数据。
- `src/stores/*`：跨页面状态。
- `server/routes/*`：路由入口，只处理请求响应衔接。
- `server/services/*`：业务逻辑、AI 调用、数据访问。
- `server/middleware/*`：鉴权、限流、错误处理。

## 错误处理策略

前端：

- API 层把非 2xx 响应转为统一错误对象。
- 页面只展示用户可理解的错误文案。
- 认证失效时统一清理本地登录状态。
- SSE 网络异常时结束 loading，并提示可重试。

后端：

- 参数错误返回 400。
- 未认证返回 401。
- 权限不足返回 403。
- 资源不存在返回 404。
- 服务异常返回 500，生产环境不暴露堆栈。

## 检查与验证

每个实施阶段至少执行：

- 前端：`pnpm check`
- 后端：`cd server && pnpm check`

涉及运行时行为时，额外启动前后端并用浏览器验证：

- 首页可访问。
- 登录页可访问。
- 天气接口可用。
- 推荐/聊天流式接口不阻塞页面。
- `/api/health` 正常返回。

## 风险与控制

1. 渐进式 TypeScript 可能暴露大量历史弱类型问题：本轮只迁移边界文件，不追求全量严格。
2. 后端参数校验可能改变现有宽松输入：先覆盖关键字段，错误文案保持友好。
3. 部署文档无法完全替代真实服务器环境：文档写通用流程，最终部署时根据 ECS 实际系统微调。
4. AI 接口依赖外部密钥：保留无 key 降级逻辑，避免演示完全不可用。
